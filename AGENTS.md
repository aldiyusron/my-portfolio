# Repository Guidelines

## Project Structure & Module Organization
- `src/` contains application code.
- `src/main.jsx` bootstraps React and renders `App`.
- `src/App.jsx` is the app entry component and currently delegates to `src/Portfolio.jsx`.
- `src/assets/` stores local media used by components.
- `public/` stores static files served as-is (for example `favicon.svg`, `icons.svg`).
- `dist/` is the production build output generated by Vite; do not edit by hand.

## Build, Test, and Development Commands
- `npm install`: install dependencies.
- `npm run dev`: start local Vite dev server with HMR.
- `npm run build`: create optimized production bundle in `dist/`.
- `npm run preview`: serve the built output locally for verification.
- `npm run lint`: run ESLint across the repo.

Use `npm run lint && npm run build` before opening a PR.

## Coding Style & Naming Conventions
- Language stack: modern JavaScript + React (JSX, ES modules).
- Use 2-space indentation and semicolon-free style to match current files.
- Prefer single quotes in new code unless surrounding context already uses double quotes.
- Components: `PascalCase` filenames and exports (for example `Portfolio.jsx`).
- Variables/functions: `camelCase`; constants: `UPPER_SNAKE_CASE`.
- Keep UI class names scoped and prefixed consistently (current pattern: `pf-*`).
- Linting is defined in `eslint.config.js` (React hooks + refresh rules enabled).

## Testing Guidelines
- There is currently no automated test suite configured.
- Minimum validation for every change: `npm run lint` and a manual browser check via `npm run dev`.
- If adding tests, use Vitest + React Testing Library and place files as `*.test.jsx` near the component under test.

## Commit & Pull Request Guidelines
- Follow concise, imperative commit messages (examples from history: `Fix favicon and reduce section spacing`, `Further reduce section gaps...`).
- Keep each commit focused on one logical change.
- PRs should include:
  - clear summary of user-visible and code-level changes,
  - linked issue/ticket when applicable,
  - before/after screenshots for UI updates,
  - confirmation that lint and build passed locally.
